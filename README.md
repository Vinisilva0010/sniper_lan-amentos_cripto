TECHNICAL DESIGN DOCUMENT (TDD) - V2.0

Projeto: Apex Genesis (Codinome: Shadow Weaver)Target: Solana (Mempool, Raydium AMM V4, Pump.fun)Arquitetura: Híbrida de Alta Frequência (Rust Bare-Metal + Python Asyncio)Tese Operacional: Rastreio de financiamento de carteiras de desenvolvedores (Cabal Tracking) combinado com Counter-Sniping via Jito Bundles.


 1. ARQUITETURA CORE E COMUNICAÇÃO (IPC)Nossa maior premissa: Latência é morte. O Python não vai encostar na rede da Solana para ler dados pesados.O Músculo (Rust): Conectado via gRPC (Yellowstone) direto num bare-metal ou VPS parruda (Helius/Alchemy). Fica responsável por mastigar os blocos binários.O Cérebro (Python/Async): Orquestra a lógica de negócio, risco e gestão de portfólio.A Ponte (ZeroMQ + MsgPack): Esqueça TCP e JSON. A comunicação entre Rust e Python será feita via ZeroMQ (ZMQ) rodando sobre Unix Domain Sockets (UDS) no padrão PUSH/PULL (Rust empurra, Python puxa). O payload será serializado em MessagePack (MsgPack).Meta de latência do IPC: 15µs a 50µs.


 2. OS 4 MOTORES (COMPONENTES DO SISTEMA)Motor 1: O Sabujo (Rust gRPC Ingestor)Missão: Encontrar o rastro do dinheiro antes do lançamento.Lógica: Escuta a rede focando no System Program (transferências de SOL). O filtro procura especificamente saídas de CEX (Binance, MEXC) ou Mixers indo para carteiras com histórico zero.Gatilho: Se detecta um padrão de cluster (ex: múltiplas carteiras virgens recebendo valores fracionados idênticos em uma janela curta), o Rust empacota o array de endereços em MsgPack e faz um PUSH via ZMQ para o Python.


Motor 2: Lógica (Modelo Matemático de Clusterização): O Event Loop do Python (Puller) recebe as carteiras financiadas e as submete a um modelo de Hard Thresholds na RAM. Um cluster de carteiras virgens só é classificado como Cabal e enviado para a UTI (Target List) se passar na seguinte equação:$$\text{Cabal}_{trigger} = (N \ge 4) \land (\Delta t \le 180) \land \left(\frac{\sigma}{\mu} \le 0.05\right) \land (S = 1)$$$N$: Número de carteiras virgens no cluster (mínimo de 4).$\Delta t$: Janela de tempo total das transferências (máximo de 180 segundos).$\frac{\sigma}{\mu}$: Coeficiente de Variação dos valores transferidos em SOL (desvio padrão $\sigma$ dividido pela média $\mu$ deve ser $\le 5\%$, indicando scripts de distribuição automatizados).$S$: Origem Comum (Boolean $1$ se os fundos vieram da mesma CEX/Mixer).


Motor 3: O Escudo Anti-Rug (Rust Atomic Simulator)Missão: Validação brutal e à prova de Honeypots modernos (O Blockpoint).Lógica: No milissegundo em que uma carteira da UTI interage com um roteador (Raydium) ou Token Program criando liquidez, o Rust trava o fluxo e roda uma Simulação Atômica (Buy -> Sell) localmente na RAM usando o estado atual da rede.Pipeline de Simulação:Forja instrução de Compra (Swap SOL -> Token).Forja instrução de Venda (Swap 100% Token -> SOL).Condições de Aborto (Kill Switch):Se a instrução de VENDA falhar (Honeypot condicional ativado).Se o retorno de SOL for menor que 95% do esperado descontando taxas da DEX (Tax/Fee escondida ou CPI de drenagem de liquidez).Se passar limpo, sinal verde para o Motor 4.


Motor 4: O Executor e a Guilhotina" e substitua o tópico Entrada (Compra Dinâmica) inteiro por este bloco aqui:Entrada (Compra Dinâmica via Jito Bundles): Envio direto para validadores via API do Jito para evitar Mempool pública. O Tip (propina) não é fixo; o bot assina o Tip Stream do Jito e calcula o valor dinamicamente baseado em um Hype Score ($H_{score}$) 100% on-chain.Cálculo do $H_{score}$: Avalia o tamanho da Cabal ($N$), a liquidez inicial adicionada em SOL ($L_{sol}$) e o spam de bots concorrentes detectados no bloco ($M_{bots}$).$$H_{score} = 1.0 + \min\left(4.0, \left( \frac{N - 4}{10} \right) + \left( \frac{L_{sol}}{20} \right) + \left( \frac{M_{bots}}{50} \right) \right)$$Cálculo do Tip Final: Baseado no Percentil 75 ($P_{75}$) da rede no momento do bloco.$$Tip_{final} = P_{75} + (0.001 \cdot H_{score})$$Circuit Breaker: Se o blockhash expirar (transação não incluída na janela alvo), o Bundle é sumariamente dropado pelo Jito. Risco de ficar "pendurado" = Zero.


 3. SOBREVIVÊNCIA E TELEMETRIA (O BUNKER)Recovery State (Tolerância a Falhas): O motor Python salva micro-estados a cada compra confirmada em um banco SQLite local configurado em modo WAL (Write-Ahead Logging) ou memória mapeada. Se o processo do Python crashar (Out of Memory, Bug), o Supervisor do SO reinicia em < 1s. Ao subir, o main.py lê o WAL. Se houver posição aberta, ele pula todo o rastreio e assume imediatamente o controle da venda (Modo de Gestão de Crise). Nunca ficamos com o bot cego e comprado.Fase 1 - Paper Trading Obrigatório: O bot rodará inicialmente com a flag PAPER_TRADING = True. Ele fará o track completo, simulará a compra, acompanhará a proporção das pools e registrará o PNL (Lucro/Perda) teórico no PostgreSQL. O deploy com capital real (SOL > 0) só será destravado após 500 lançamentos simulados com curva de equidade positiva validada por nós.
